#  BRD-017 — Reproducible PHP 5.5 test environment 🏗️

## Status

Planned

## Goal

Provide a reproducible PHP 5.5 command-line environment for running the project's PHP validation workflow.

The production NOC currently runs on PHP 5.5, while development machines may use newer PHP versions. This creates a risk that tests and syntax checks pass locally but fail after deployment due to differences in language features or runtime behaviour.

This slice establishes a canonical Docker-based PHP 5.5 environment that every developer can use to validate the project before deployment.

---

## Design

### Docker image

Create:

```text
tests/docker/php55/Dockerfile
```

The Docker image shall provide a reproducible PHP 5.5 CLI environment suitable for validating the project.

The image shall contain only the tooling required to execute the project's validation workflow.

---

### Validation workflow

The PHP validation workflow shall execute entirely inside the Docker container.

Initially this includes:

- all PHP unit tests;
- syntax checking of all PHP source files using `php -l`.

The workflow shall be executable using a single documented command.

---

### Developer workflow

Developers shall not need a local PHP 5.5 installation.

The Docker image becomes the canonical environment for validating PHP compatibility with production.

Local PHP installations may still be used for convenience, but the Docker environment is considered authoritative.

---

## Acceptance Criteria

- [ ] `tests/docker/php55/Dockerfile` exists.
- [ ] The Docker image builds reproducibly.
- [ ] The image provides a working PHP 5.5 CLI environment.
- [ ] All PHP tests execute successfully inside the container.
- [ ] All project PHP files are validated using `php -l` inside the container.
- [ ] The complete validation workflow is executable using a single documented command.
- [ ] The validation workflow does not depend on the developer's locally installed PHP version.
- [ ] The Docker image contains no deployment-specific configuration or secrets.

---

## Notes

This slice improves confidence rather than functionality.

Its purpose is to make production compatibility reproducible during development so that PHP version incompatibilities are detected before deployment.

The Docker environment intentionally models only the PHP runtime required for testing. It is **not** intended to reproduce the complete production hosting environment.
