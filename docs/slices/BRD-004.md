# BRD-004 — Implement Bredland Heartbeat

## Status

Complete

## Goal

Implement a generic telemetry client for the Bredland Raspberry Pi using the telemetry endpoint introduced in BRD-003.

This slice also establishes automated behavioural tests for the shared PHP endpoint before introducing a second production client.

---

## Design

### Bredland telemetry client

Implement a heartbeat client that periodically POSTs telemetry to the generic endpoint.

Initially mirror the information currently provided by the MikroTik where practical, allowing future host-specific fields to be added without endpoint changes.

---

### Endpoint behavioural tests

Extract the endpoint's request validation and record-building logic into small, testable PHP functions.

Implement a lightweight PHP test harness exercising the endpoint behaviour without requiring a full HTTP server.

The tests should verify at least:

* valid host/token accepted
* bad token rejected
* unknown host rejected
* reserved fields rejected
* advertised fields copied into the stored record
* stored JSON schema remains version 1

The test infrastructure should become the preferred way of validating future endpoint changes.

---

## Acceptance Criteria

* Bredland posts successful telemetry to the generic endpoint. ✅
* Existing MikroTik telemetry remains unaffected. ✅
* Endpoint behavioural tests pass locally and in CI. ✅
* Endpoint behaviour is verified without manual smoke testing. ✅
