# BRD-015 — Boolean heartbeat fields 🍕

## Status

Complete

## Goal

Introduce `boolean` as a first-class heartbeat value type and use it to expose whether a MikroTik RouterOS software update is available.

This slice intentionally keeps the new value type small and focused. Its purpose is to establish the contract for boolean telemetry values before they are used by future dashboard features.

The slice is informational only. It does **not** perform RouterOS updates or generate dashboard notifications.

---

## Design

### MikroTik telemetry

The MikroTik heartbeat shall expose a new boolean field:

```
"update_available": true
```

or

```
"update_available": false
```

The value shall be derived from RouterOS package update information by comparing the installed RouterOS version with the latest available version.

---

### Dashboard

This slice introduces `boolean` as a supported `valueType`.

The dashboard shall:

- support `boolean` in client descriptions;
- validate that boolean fields contain only boolean values;

No special dashboard behaviour is introduced as part of this slice.

---

## Acceptance Criteria

- MikroTik heartbeats include an `update_available` boolean field. ✅
- `boolean` is supported as a dashboard `valueType`. ✅
- Dashboard validation rejects non-boolean values for boolean fields. ✅
- The MikroTik client telemetry drawer includes the update status. ✅
- Existing client types continue to function unchanged. ✅

---

## Notes

This slice intentionally introduces only the `boolean` value type and a single real-world use case.

Future slices may build upon this foundation to support richer dashboard behaviour, including notifications, client-defined health, LED colours, badges, and other declarative presentation rules.

The `boolean` value type is introduced now because it has earned its keep through a genuine operational need, rather than as a speculative extension to the data model.
