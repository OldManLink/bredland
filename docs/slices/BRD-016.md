#  BRD-016 — Declarative dashboard rules 🏗️

## Status

Planned

## Goal

Introduce a generic rules engine that allows client descriptions to derive dashboard behaviour from heartbeat values.

The dashboard currently renders heartbeat values but contains no mechanism for deriving higher-level behaviour from them. This slice introduces a declarative rules engine that evaluates heartbeat facts and produces dashboard actions without requiring client-specific PHP.

The first implementation intentionally supports only a small set of conditions and actions. Future slices may extend the same mechanism as new requirements emerge.

---

## History

This capability originally formed part of **BRD-007 — Self-describing clients**.

During implementation it became clear that declarative rule evaluation was a substantial architectural feature in its own right, so it was deferred in order to keep BRD-007 focused and deliverable.

---

## Principles

- Heartbeats contain facts.
- Client descriptions define presentation.
- Rules derive behaviour from facts.
- Dashboard PHP contains no client-specific logic.

---

## Design

### Rules

A client description may contain a `rules` array.

Each rule consists of:

- a `when` condition;
- one or more `then` actions.

Example:

```json
{
  "rules": [
    {
      "when": {
        "field": "update_available",
        "equals": true
      },
      "then": [
        {
          "type": "notification",
          "text": "RouterOS {{latest_version}} is available."
        }
      ]
    }
  ]
}
```

Rules are evaluated after the latest heartbeat has been loaded.

Each rule whose condition evaluates to true produces the actions declared by its `then` clause.

---

### Initial rule support

The initial implementation intentionally keeps the rule language small.

Supported condition operators:

- `equals`

Additional operators (such as `<`, `>`, `<=`, `>=`, etc.) may be introduced by future slices as genuine use cases arise.

---

### Initial actions

The initial implementation supports:

- `notification`

Future slices may introduce additional actions, including:

- `health`
- `led`
- `badge`
- `highlight`
- `icon`

These are **not** part of this slice.

---

### Notifications

Notification actions are collected for each client.

If one or more notifications exist:

- the dashboard displays a notification badge;
- the badge indicates the number of active notifications;
- selecting the badge displays the active notification messages.

Notifications are derived entirely from the latest heartbeat.

No notification state is stored.

Notifications disappear automatically when their conditions are no longer satisfied.

Notification text may interpolate heartbeat fields using the existing template syntax.

---

## Acceptance Criteria

- [ ] Client descriptions may contain declarative rules.
- [ ] Rules are evaluated generically from heartbeat values.
- [ ] `equals` conditions are supported.
- [ ] `notification` actions are supported.
- [ ] Notification messages may interpolate heartbeat fields.
- [ ] Multiple notifications are supported.
- [ ] Notifications disappear automatically when their conditions become false.
- [ ] Dashboard PHP remains independent of individual client types.

---

## Notes

This slice intentionally introduces a generic rule engine rather than a collection of special-case dashboard features.

The long-term goal is for client descriptions—not dashboard PHP—to define how telemetry influences presentation.

Each new dashboard behaviour should, where practical, be introduced as a new declarative rule action rather than as client-specific code.
