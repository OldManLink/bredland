# BRD-007 — Self-describing clients

## Status

Complete

## Goal

Move dashboard knowledge out of PHP code and into per-client description files.

The telemetry endpoint remains generic in the sense that it contains no hard-coded knowledge of individual hosts. Each monitored host instead has a complete heartbeat schema describing the fields that may be stored and their required runtime types.

The endpoint uses that schema to convert incoming POST values into correctly typed heartbeat records before appending them to the host’s JSONL stream.

The dashboard uses separate client description files to define which heartbeat fields should be displayed and how they should be formatted.

After this slice is complete, adding a new monitored client shall require:

1. A heartbeat schema.
2. A dashboard client description.
3. A client that posts telemetry conforming to the schema.

No client-specific dashboard PHP changes shall be necessary.

---

## Design

### Heartbeat schemas

A new `schemas/` directory contains one complete heartbeat schema for each monitored host.

Initially:

```text
schemas/
    bredland.json
    mikrotik.json
```

Each schema describes the complete stored heartbeat, including server-generated fields such as `schema`, `ts`, `host`, and `remote_addr`.

Every schema field defines exactly one of:

* `const` — the field must have the declared constant value.
* `valueType` — the incoming value must be converted to and stored with the declared runtime type.

The initially supported value types are:

* `integer`
* `float`
* `string`

The telemetry endpoint loads the schema corresponding to the authenticated host and uses it to construct the stored heartbeat. Missing fields, undeclared fields, invalid values, or failed type conversions are rejected.

### Client descriptions

A `clients/` directory contains one JSON description file for each dashboard client.

Initially:

```text
clients/
    bredland.json
    mikrotik.json
```

Each client description defines:

* the host and display title,
* card ordering,
* which heartbeat fields should be displayed,
* the expected runtime type of each displayed field, and
* any dashboard formatter to apply.

Client descriptions are dashboard views over the complete heartbeat schemas. Every displayed field must exist in the corresponding schema and declare the same `valueType`.

### Generic dashboard

The dashboard no longer contains client-specific knowledge.

Instead, it:

1. Discovers all client descriptions in `clients/`.
2. Reads the latest heartbeat for each described client.
3. Validates heartbeat values against the declared runtime types.
4. Renders each dashboard card from its client description.

Malformed client descriptions are skipped defensively rather than taking down the dashboard.

### Dashboard capabilities

Client descriptions may reference named dashboard formatters.

The exported formatter registry defines:

* which formatter functions are available, and
* which value types each formatter accepts.

Tests verify that exported formatter functions exist, accept exactly one parameter, and are compatible with the value types declared by client descriptions.

### Production heartbeat fixtures

The heartbeat fixtures under `tests/fixtures/heartbeats/` are complete heartbeat objects copied from production.

Tests validate each fixture against its complete heartbeat schema, including:

* exact field equality,
* constant values, and
* runtime value types.

This ensures that production telemetry satisfies the same contract used by the endpoint and dashboard.

## Acceptance Criteria

* The telemetry endpoint loads a complete heartbeat schema for the authenticated host. ✅
* Incoming POST values are converted to the runtime types declared by the schema. ✅
* Invalid, missing, or undeclared heartbeat fields are rejected. ✅
* Stored heartbeats contain correctly typed JSON values. ✅
* Production heartbeat fixtures are validated completely against their schemas. ✅
* The dashboard discovers client descriptions automatically from `clients/`. ✅
* No dashboard PHP code contains client-specific rendering logic. ✅
* MikroTik and Bredland are rendered entirely from their respective client description files. ✅
* Client descriptions are validated against their corresponding heartbeat schemas. ✅
* Client descriptions may reference exported dashboard formatters by name. ✅
* Formatter existence, signature, and value-type compatibility are validated during testing. ✅
* Malformed client descriptions do not take down the dashboard. ✅
* Adding a new client requires no modification to dashboard PHP code. ✅
