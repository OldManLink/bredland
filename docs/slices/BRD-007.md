# BRD-007 — Self-describing clients

## Status

Complete

## Goal

Move dashboard knowledge out of PHP code and into per-client description files.

The telemetry endpoint is intentionally generic. It accepts heartbeat data from clients, adds the common metadata required by the server, and stores each heartbeat in a per-host JSONL stream without interpreting the payload.

The dashboard, however, currently contains hard-coded knowledge about every monitored client. It knows which clients exist, which fields they report, how those fields should be formatted, and how each dashboard card should be constructed.

This slice introduces per-client description files that encapsulate this knowledge, allowing the dashboard to become a generic renderer.

After this slice is complete, adding a new client to the dashboard shall require only the deployment of the client and its corresponding file in `clients/`; no dashboard PHP changes shall be necessary.

---

## Design

### Client descriptions

A new `clients/` directory shall contain one JSON description file for each monitored client.

Initially this directory shall contain:

```
clients/
    bredland.json
    mikrotik.json
```

Each client description defines everything the dashboard needs in order to render that client's card, including field definitions, formatting instructions, and presentation metadata.

### Generic dashboard

The dashboard shall no longer contain client-specific knowledge.

Instead, it shall:

1. Discover all client descriptions in `clients/`.
2. Read the latest heartbeat for each described client.
3. Render a dashboard card according to the client description.

The dashboard shall not require any knowledge of individual clients beyond the capabilities it exposes through its rendering library.

### Dashboard capabilities

Client descriptions may reference named dashboard capabilities such as value formatters or click actions.

These capabilities form the contract between the client descriptions and the dashboard implementation.

---

## Acceptance Criteria

* The dashboard discovers client descriptions automatically from `clients/`. ✅
* No dashboard PHP code contains client-specific rendering logic. ✅
* MikroTik and Bredland are rendered entirely from their respective client description files. ✅
* Client descriptions may reference dashboard capabilities by name. ✅
* Client descriptions are validated against the dashboard capabilities during testing. ✅
* Adding a new client requires no modification to the dashboard PHP code. ✅
